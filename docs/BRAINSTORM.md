# Dayline 灵感碰撞 × 修改方案

> 生成方式：AI 读完项目全部代码后，基于现有架构和产品理念的推演。
> 不是需求文档，是"如果是我，接下来会做什么"的思考。
> 每一条都考虑了：现有架构能不能承载、改动范围多大、是否违背 local-first 原则。

---

## 一、现状一句话

Dayline 有一个非常好的骨架：自然语言输入 → 解析 → 分类落库 → 四视图消费。但「解析后的数据」和「用户真正想从中获得的价值」之间，还有一大段空白。

---

## 二、碰撞出的方向

### 🔥 方向 1：把 QuickInputBar 变成真正的"一键记录"

**现状**：首页的 `QuickInputBar`（saveImmediately 模式）已经能打字保存，但没有充分利用。

**想法**：让首页输入框成为一个"万能入口"——不只是打字，而是支持：
- 长按 = 语音记录（录音 → 本地保存音频 → 语音转文字 → 解析 → 分类）
- 输入 `#跑步 30分钟` 直接解析为打卡记录
- 输入 `¥35 午餐` 直接解析为消费
- 输入 `待办 明天交报告` 直接解析为待办

**技术可行性**：`LuiLiteParser` 已经支持这些格式，只是目前 QuickInputBar 的 saveImmediately 路径把解析结果全部当 memo 存了。修改 `_save()` 逻辑即可启用。

**改动范围**：小。改 `QuickInputBar` 的 `_submitText()` 方法 + `RecordNotifier.saveInput()` 的路由逻辑。

**建议优先级**：🔴 高。这是"低摩擦"的核心理念的具体落地。

---

### 🔥 方向 2：语音记录（完全本地）

**现状**：麦克风图标只有视觉，无功能。

**想法**：
```
长按麦克风 → 录音（本地 .m4a）
  → 松手 → 保存音频文件到 app documents
    → 同时尝试语音转文字（本地优先）
    → 成功 → 文本填入输入框 → 可手动编辑 → 保存 Entry（关联音频文件）
    → 失败 → 创建纯语音 Entry（rawVoicePath 不为空，displayText 显示"语音记录 [未识别]"）
```

**隐私策略**：
- 默认使用系统内置语音识别（Android SpeechRecognizer / iOS SFSpeechRecognizer），部分设备支持离线
- 提供选项切换到完全离线方案（如 whisper.cpp 的 Flutter 封装）
- 不上传音频到任何第三方

**技术选型建议**：
- 录音：`record` 包（纯 Dart）
- 语音识别：先用 `speech_to_text` 包（系统内置引擎），后续可扩展到 Vosk/whisper

**改动范围**：中。需要新增 `SpeechService` 抽象层 + 录音 UI 组件 + 权限配置。

**建议优先级**：🔴 高。这是"低摩擦记录"最重要的一块拼图。

---

### 🔥 方向 3：AI 附加层（完全可选的、可重算的、不覆盖原始数据的）

**现状**：`StatusInsightCard` 和 Review 页的摘要都是静态 mock 数据。"AI"在代码中不存在。

**想法**：构建一个分层的 AI 系统：

```
AiService (abstract)
  ├── MockAiService      — 返回固定 mock（开发/测试用）
  ├── LocalAiService     — 本地模型（ML Kit / llama.cpp / mediapipe）
  └── CloudAiService     — 云端模型（明确隐私边界，默认关闭）
```

**三阶段 AI 能力**：

| 阶段 | 能力 | 触发时机 | 数据出本机？ |
|------|------|---------|------------|
| 1 | 自动标签建议 | 保存记录后异步触发 | 取决于 AiService 实现 |
| 2 | 每日/每周摘要 | 用户打开 Review 页时触发 | 同上 |
| 3 | 个人数据库问答 | 用户自然语言问"我上个月跑步了几次" | 同上 |

**关键技术设计**：
- AiEnrichment 表已经设计好了（见数据模型建议），AI 结果存在独立表里
- `AiEnrichment.modelName` 记录用的是哪个模型，方便追溯和重算
- 原始记录 (`Entry.rawText`) 永远不传给 AI，只传 `displayText`（用户可编辑的版本）
- 如果用户删除了 AI 结果，可以随时重新生成

**改动范围**：中到大。需要新建 `lib/core/ai/` 目录、AiService 接口、本地/云端实现、数据库表扩展。

**建议优先级**：🟡 中。先做好抽象层和 Mock 实现（1-2 天），云端接入选型可以后做。

---

### 🔥 方向 4：搜索（本地全文搜索）

**现状**：无搜索。用户记录多了之后只能通过 Timeline 翻页。

**想法**：利用 SQLite FTS5（全文本搜索）做本地搜索。

```sql
CREATE VIRTUAL TABLE records_fts USING fts5(
  content, tags, title,
  content='records',
  content_rowid='id'
);
```

搜索流程：
1. 用户在顶部搜索栏输入关键词
2. `SELECT * FROM records_fts WHERE content MATCH ?` → 模糊匹配
3. 结果按时间倒序排列
4. 支持按类型、标签、日期范围过滤

**改动范围**：中。需要新建 FTS 表 + 搜索 UI + 搜索结果页。

**建议优先级**：🟡 中。搜索是个人数据库的基础能力，越早做越好，因为数据越多越需要。

---

### 🔥 方向 5：数据导入 + 备份恢复

**现状**：只有导出（Markdown/JSON），没有导入。

**想法**：
- 从 JSON 文件导入（与导出格式一致，保证可往返）
- 导入时做冲突检测（日期 + 内容哈希去重）
- 备份：自动在 app documents 目录生成 `.dayline_backup/` 定期快照

**改动范围**：中。`ExportService` 的反向操作 + 文件选择器 UI。

**建议优先级**：🔴 高。个人数据库 App 数据导入是刚需（换手机、从别的 App 迁数据）。

---

### 🔥 方向 6：隐私模式

**现状**：数据库设计里有软删字段 (`isDeleted`, `isPrivate`)，但 UI 没做。

**想法**：
- 首页预览行：私密记录不显示内容，只显示"🔒 私密记录"
- Timeline：私密记录模糊处理，需要 Face ID/指纹/密码验证后查看
- 导出：可选择是否包含私密记录
- 快速切换：AppBar 上增加隐私模式开关

**改动范围**：小。利用已有的 `Entry.isPrivate` 字段 + UI 条件渲染。

**建议优先级**：🟡 中。实现简单，对隐私敏感的日记用户来说是强需求。

---

### 🔥 方向 7：标签系统从"附属于记录"升级为"一等公民"

**现状**：标签以 JSON 数组字符串存在 `records.tags` 字段里。没有标签管理页面，没有标签统计。

**想法**：
- 新建 `tags` 表（独立存储标签名、颜色、图标、父标签）
- 新建 `entry_tags` 关联表（多对多）
- 标签管理页：查看所有标签、合并、重命名、设置颜色
- 标签筛选：在 Timeline 和搜索中按标签过滤
- 标签云：在 Review 页可视化展示最常用标签

**数据库变更**：
```sql
tags (id, name, color, icon, parent_id, is_auto, created_at)
entry_tags (entry_id, tag_id, source, confidence, tagged_at)
```

**改动范围**：中到大。需要数据库 Migration + 新页面 + 现有记录页的标签选择器改造。

**建议优先级**：🟡 中。标签是连接"记录"和"回顾"的桥梁。

---

### 🔥 方向 8：个人量化→从 Entry 中自动提取 Metric

**现状**：`trackers` 和 `tracker_logs` 表已经存在，但需要用户手动创建打卡项和手动记录数据。

**想法**：如果用户写了"今天跑步 30 分钟"，解析器识别为 tracker 类型，**自动**：
1. 检查是否已有名为"跑步"的 tracker → 没有则创建
2. 从文本中提取时长 `30分钟`
3. 自动创建 tracker_log
4. 在 Today 页的 `TodayTrackersCard` 中自动显示

这个其实 `LuiLiteParser` 已经能识别 tracker 类型和时长，但 `RecordNotifier._saveTrackerLog()` 的逻辑需要完善。

**改动范围**：小。主要是 `RecordNotifier` 的修复和增强。

**建议优先级**：🔴 高。这是让"追踪"从手动变成自动的关键一步。

---

### 🔥 方向 9：Weekly / Monthly Review 页面

**现状**：Review 只有每日回顾。

**想法**：增加周回顾和月回顾：
- WeekReview：本周运动总时长、消费总额、完成待办数、情绪趋势、高频标签
- MonthReview：月度趋势图（用 `fl_chart` 做轻量图表）、习惯连续性、消费分布饼图

**改动范围**：中。需要扩展 Review providers 支持时间范围参数 + 新增图表依赖。

**建议优先级**：🟢 低。先做好日回顾，周/月可以后续迭代。

---

### 🔥 方向 10：无网络环境下的 AI 能力（On-device ML）

**现状**：无 AI 代码。

**想法**：利用 Google ML Kit 或 MediaPipe on-device：
- **文本分类**：本地模型对记录进行自动分类（工作/生活/健康）
- **情绪分析**：本地 NLP 模型提取情绪倾向
- **实体提取**：识别人名、地点、数字

这些完全不依赖网络，数据不出设备。

**技术路线**：
- `google_mlkit_text_recognition` — 文字识别
- `google_mlkit_entity_extraction` — 实体提取
- 或者用 `mediapipe_text_classifier` 做文本分类

**改动范围**：大。新依赖 + 模型文件可能较大 + 需要 AiService 的 `LocalAiService` 实现。

**建议优先级**：🟢 低。先做好本地搜索和手动标签，on-device ML 是锦上添花。

## 四、架构升级建议（不紧急但重要）

### 4.1 引入类型化的 Data Model

**现在**：所有 DB 数据以 `Map<String, Object?>` 传递。

**建议**：在 Phase 1 中逐步引入 model class（可以手写，不一定要 freezed）。至少：
- `Record` → 替代原始 Map
- `Todo` → 同上
- `Tracker` / `TrackerLog` → 同上

**好处**：编译时类型检查、IDE 自动补全、重构安全。

### 4.2 统一类型映射表

**现在**：7 种记录类型的 label/color/icon 在 3 个文件中重复定义。

**建议**：在 `ParsedInputType` 上扩展为：
```dart
enum ParsedInputType {
  memo(label: '备忘', icon: Icons.notes, color: AppColors.memo),
  todo(label: '待办', icon: Icons.check_circle, color: AppColors.todo),
  // ...
}
```

**改动**：一次性，影响约 3-5 个文件。

### 4.3 从 sqflite 迁移到 Drift

**现在**：裸写 SQL 字符串。

**建议**：Phase 2 完成后考虑迁移。Drift 提供：
- 类型安全的查询
- 自动 Migration 管理
- 编译时 SQL 检查
- Stream-based 数据变更通知（可替代 DataVersionNotifier）

**代价**：学习曲线 + 代码生成步骤 + 迁移工作量。

**时机**：不要在功能开发中做迁移。选一个 Phase 之间的空窗期。

## 五、不做的事情（明确拒绝）

这些想法可能看起来合理，但与项目理念冲突：

| 想法 | 拒绝理由 |
|------|---------|
| 云端同步 (Firebase/Firestore) | 违反 local-first 原则。用户数据不应该强制上云。如果做，必须作为可选功能且默认关闭。 |
| 社交分享/社区功能 | Dayline 是私人日记，不是社交媒体。 |
| 游戏化/积分/等级 | 量化自我 ≠ 游戏化。严肃的自我追踪不需要徽章和等级。 |
| AI 自动发朋友圈 | 把私密日记送去社交平台是隐私灾难。 |
| 引入重量级状态管理重构 | 当前 Riverpod 用得很好，不要为换而换。 |
| 一次性大重构 | 不管代码有多少槽点，分阶段改。功能开发和新架构可以并行渐进推进。 |
| iOS 专属功能（iCloud sync 等） | iOS 目录都还没建。先做好 Android，iOS 适配合时再做。 |

---

## 六、总结

Dayline 目前是一个 **架构清晰、骨架完整、测试覆盖良好** 的项目。代码质量在个人项目里属于上乘。

当前最大的机会不是"重写什么"，而是：

1. **把 QuickInputBar 变成真正的万能入口**（打字 + 语音 + 自动分类）
2. **把标签从 JSON 字符串升级为一等公民**（独立表 + 管理页面 + 筛选）
3. **构建 AiService 抽象层**（先做规则引擎，再考虑接模型）
4. **补上搜索和导入**（个人数据库的基本功）

这四件事做完，Dayline 就从一个"能用的记录工具"变成了一个"离不开的个人数据库"。
