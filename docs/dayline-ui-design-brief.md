# Dayline UI 设计简报

## 1. 产品一句话

Dayline 是一款本地优先的个人生活记录应用，帮助用户用一句话快速记录今天发生的事、待办、打卡、专注、消费、身体状态和睡眠，并在 Today、Timeline、Review 中把这些碎片整理成当天的概览、时间线和复盘。

它不是社交产品，也不是重型效率工具。Dayline 更像一个安静、可信、长期陪伴的私人日记和生活仪表盘。

## 2. 产品定位

Dayline 的核心价值是降低记录成本，同时让用户能在事后看见自己的生活脉络。

用户不需要先选择复杂分类，也不需要填表。理想体验是：打开 App，输入一句自然语言，系统自动识别记录类型，再把内容放到合适的位置。记录完成后，Today 提供今日状态，Timeline 按时间还原一天，Review 帮用户做总结和导出。

设计上应避免过度游戏化、社交化和营销感。界面需要克制、清晰、温和，有“我愿意每天打开它”的亲近感，同时保持高频记录场景下的效率。

## 3. 目标用户

### 3.1 核心用户

1. 自我管理型用户  
   关心时间、习惯、专注、消费和身体状态，希望通过记录理解自己。

2. 轻量日记用户  
   不想写长篇日记，但愿意用一句话留下每天的重要片段。

3. 学生、自由职业者、独立开发者、知识工作者  
   日常节奏分散，既需要待办和专注记录，也需要复盘和时间线回看。

4. 隐私敏感用户  
   希望数据保存在本地，不依赖登录、云同步、广告或追踪 SDK。

### 3.2 用户心理

用户打开 Dayline 时通常不是为了“管理一个系统”，而是为了快速放下脑中的一件事。界面应该让用户感觉轻、快、可信，不要让用户先面对复杂配置。

用户长期使用后，希望看到“我这一天/一周/一段时间是怎么过的”。因此数据展示不应只是冷冰冰的数字，而要能帮助用户建立生活感和连续性。

## 4. 当前产品功能

### 4.1 四个底部 Tab

1. Today  
   今日首页，展示日期、快速输入、今日概览、时间进度、今日打卡、今日待办、最近记录。

2. Record  
   记录页，强调快速输入和解析确认。用户输入一句话后，系统会展示解析类型、置信度、标签、时间和元数据，用户可以确认保存、取消或转为备忘。

3. Timeline  
   时间线页，按日期浏览当天所有事件。支持前一天/后一天切换和回到今天。不同记录类型有不同颜色、图标和类型标签。

4. Review  
   复盘页，按日期查看当日总结、数据概览、热门标签，并支持导出 Markdown 和 JSON。

### 4.2 记录类型

当前自然语言解析支持以下类型：

- 备忘 memo
- 待办 todo
- 打卡 tracker
- 专注 focus
- 消费 expense
- 身体 body
- 睡眠 sleep

解析能力包括：

- 识别中文和英文关键词
- 提取 `#标签`
- 提取时间，如 `09:30`、`9点`、`9点半`
- 提取消费金额
- 提取体重等身体数值
- 提取专注时长

### 4.3 本地优先

当前数据使用本地 SQLite。产品不包含网络登录、广告、分析 SDK 或追踪能力。这是重要的产品信任点，设计中可以体现“私人、本地、低打扰”的气质，但不要用过多说明文字占据主流程。

## 5. 当前视觉基调

当前版本使用 Material 3 风格，整体是浅色、温和、卡片式的信息面板。

现有色彩：

- 主色：深青绿色 `#2F6F73`
- 辅助色：暖金色 `#E2A93B`
- 强调色：柔和砖红 `#C76755`
- 文字主色：近黑青灰 `#1C2526`
- 次级文字：灰青色 `#667679`
- 背景：暖白米色 `#F7F5EF`
- 卡片：白色 `#FFFFFF`

当前观感优点：

- 温和、不刺眼，适合生活记录。
- Today 首页结构清晰，用户能快速看到“今天”。
- 底部导航直观，四个主模块易理解。
- 类型颜色已经初步建立：待办蓝、打卡绿、专注橙、消费红、身体紫、睡眠蓝紫。

当前需要设计师优化的方向：

- 信息密度需要更精致，避免卡片过多导致首页像简单堆叠。
- 快速输入应成为核心入口，但现在视觉权重仍偏普通。
- 记录解析确认页需要更强的信任感，让用户一眼知道系统理解了什么。
- Timeline 需要更有“生活时间线”的节奏感，而不是普通列表。
- Review 需要从“数据卡片”升级为“有洞察的复盘页面”。
- 中英文混排、数字、图标、空状态需要统一规范。

## 6. 设计原则

### 6.1 快速记录优先

用户最常做的动作是输入一句话并保存。设计应让输入入口在 Today 和 Record 中都足够明显，交互路径短，确认状态清楚。

### 6.2 不打扰

Dayline 不应给用户制造压力。不要使用过多红色警告、强排名、连续打卡焦虑、夸张成就系统。数据呈现要鼓励回看，而不是评判用户。

### 6.3 可长期使用

视觉要耐看，不要过度依赖潮流渐变、大面积装饰、复杂插画。重点应是排版、层级、留白、微妙色彩和状态反馈。

### 6.4 私密可信

产品是个人生活数据容器。界面应该像一个可靠的私人空间：清楚、稳定、低噪音。导出、删除、保存等操作要有明确反馈。

### 6.5 数据有温度

Today 和 Review 的数字不只是指标。设计可以用日期、时间进度、简短总结、标签和记录片段来营造“这就是我的一天”的感觉。

## 7. 页面设计要求

### 7.1 Today 首页

目标：让用户在 3 秒内知道今天的状态，并可以立即记录。

建议结构：

- 顶部日期和一句轻量文案，保持生活感。
- 快速输入区作为首要操作，应比普通表单更像“写下一句话”的入口。
- 今日概览展示关键数据：记录数、待办完成、打卡、专注时长。
- 时间进度可保留，但视觉上不要喧宾夺主。
- 今日打卡和今日待办应支持快速完成。
- 最近记录应让用户确认“刚才保存成功”。

设计重点：

- 输入区可以采用底部贴近手指的布局，或在 Today 中保持高优先级卡片。
- 首页卡片之间需要明确主次，不要所有卡片同等重量。
- 数字卡片应更轻，避免过度块状。
- 空状态要自然，不要让用户感到“今天还没做够”。

### 7.2 Record 记录页

目标：专注于输入、解析、确认保存。

核心流程：

1. 用户输入一句话。
2. 系统自动解析类型、时间、标签和元数据。
3. 用户看到预览。
4. 用户确认保存、取消或转为备忘。

设计重点：

- 输入框应支持长文本，但默认适合一句话记录。
- 解析预览要像“系统理解结果”，而不是表单。
- 置信度不要设计得太技术化。可以弱化为“已识别为：待办/消费/专注”等。
- 类型切换或纠错未来可能会增加，当前至少要保留“转为备忘”的位置。
- 保存成功需要有轻量反馈，并能让用户继续快速记录下一条。

示例输入：

- `todo 明天交报告 #工作`
- `9点半 跑步 30min #健康`
- `午餐 38元 #消费`
- `体重 68.5kg`
- `今天读完一章书 #阅读`

### 7.3 Timeline 时间线

目标：帮助用户回看一天发生了什么。

当前能力：

- 按日期查看。
- 前后日期切换。
- 按时间展示不同类型事件。
- 每条记录有图标、时间、内容、描述、标签和类型色。

设计重点：

- 时间线要有连续感，线条和节点应轻，不要像项目管理工具。
- 同一类型记录要有稳定颜色和图标。
- 记录内容是主角，类型标签和时间是辅助信息。
- 日期切换要简单，适合快速浏览前后几天。
- 空状态要鼓励用户开始记录，不要显得系统空荡。

### 7.4 Review 复盘

目标：把一天的数据整理成有用的总结。

当前能力：

- 日期切换。
- 当日复盘文字。
- 数据概览：记录、待办完成、打卡、专注、消费、活跃时段。
- 热门标签。
- 导出 Markdown / JSON。

设计重点：

- Review 应比 Today 更安静、更总结性。
- 复盘文字应有更高视觉权重，像当天的摘要。
- 数据网格要易扫读，但不要像后台仪表盘。
- 导出功能是工具操作，视觉层级应低于复盘内容。
- 没有数据时，应引导用户去记录，而不是展示空白报表。

## 8. 组件与状态

设计师需要覆盖以下组件状态：

- 底部导航：选中、未选中。
- 快速输入：空、输入中、可发送、保存中、错误。
- 解析预览：不同记录类型、带标签、带时间、带金额/体重/时长等元数据。
- 今日数据卡：有数据、无数据、加载中。
- 待办：未完成、已完成、点击切换。
- 打卡：未打卡、已打卡。
- Timeline：有记录、无记录、跨类型记录混排。
- Review：有复盘、无数据、导出成功、导出失败。
- 深色模式：当前代码已支持 light/dark theme，需要设计对应规范。

## 9. 信息架构建议

底部导航继续保留四项：

- Today：看今天和快速记录。
- Record：专注记录和解析确认。
- Timeline：按时间回看。
- Review：总结、洞察、导出。

Today 和 Record 都有输入入口，但角色不同：

- Today 的输入入口是高频快捷入口。
- Record 是更完整的记录工作台。

设计时要避免两个页面看起来重复。Today 应偏综合概览，Record 应偏专注输入。

## 10. 视觉方向建议

### 10.1 推荐方向

安静的生活仪表盘 + 温和的私人日记。

关键词：

- 温和
- 可信
- 轻量
- 清晰
- 有生活感
- 不社交
- 不焦虑
- 长期陪伴

### 10.2 可参考的质感

- iOS Journal 的私人感，但不要过度图片化。
- Notion/Obsidian 的低噪音信息组织，但要更移动端、更温暖。
- Apple Health 的数据清晰度，但不要医疗化。
- Day One 的日记感，但 Dayline 更强调快速记录和结构化数据。

### 10.3 不建议方向

- 不要做成任务管理软件：过多 checkbox、看板、优先级会偏离生活记录。
- 不要做成社交产品：避免动态流、头像、点赞、分享优先。
- 不要做成过度可爱或游戏化产品：连续打卡、奖章、强刺激颜色会破坏长期记录的安静感。
- 不要做成重型数据后台：过多图表会让用户觉得记录成本变高。

## 11. 设计交付物

建议前端设计师交付：

1. 移动端主流程高保真稿  
   Today、Record、Timeline、Review 四个 Tab。

2. 关键状态稿  
   空状态、加载中、保存中、错误、解析预览、导出反馈。

3. 组件规范  
   颜色、字体、间距、圆角、卡片、按钮、输入框、类型标签、图标、底部导航。

4. 深色模式  
   至少覆盖四个主页面和核心组件。

5. 记录类型视觉系统  
   为 memo、todo、tracker、focus、expense、body、sleep 定义图标、颜色、标签样式和在卡片/时间线中的表现方式。

6. 响应式说明  
   主要面向手机，但需要说明小屏手机、普通手机和大屏手机上的布局变化。

## 12. 成功标准

设计完成后，应满足：

- 用户能在首次打开时理解这是一个个人生活记录 App。
- 用户能在 3 秒内找到输入入口。
- 用户能理解输入一句话后系统解析出了什么。
- Today 不拥挤，重点清楚。
- Timeline 有时间流动感，记录可读。
- Review 有总结感，而不是普通统计页。
- 整体视觉温和、克制、耐看。
- 深色模式不只是反色，而是同样舒适。
- 所有中文文本、数字、图标和状态在移动端不拥挤、不重叠。

## 13. 当前实现约束

当前技术实现为 Flutter + Material 3 + Riverpod + GoRouter + SQLite。设计可以提出新的视觉样式，但需要尽量尊重移动端 Flutter 可实现性。

当前已有主题 token：

- 颜色集中在 `AppColors`
- 间距集中在 `AppSpacing`
- 字体集中在 `AppTypography`
- 页面使用 Material `Card`、`NavigationBar`、`IconButton`、`FilledButton`、`OutlinedButton`、`Chip`、`ActionChip`

如果设计需要明显改变视觉系统，建议同步给出 token 级规范，方便前端实现。

## 14. 给设计师的核心问题

设计时请重点回答：

1. Dayline 的第一屏如何同时做到“快速记录”和“今日概览”？
2. 一句话解析结果如何表现，才能让用户信任但不觉得技术化？
3. Timeline 如何体现生活连续性，而不是普通列表？
4. Review 如何呈现温和的洞察，而不是制造自我管理焦虑？
5. 七种记录类型如何形成统一但可区分的视觉系统？
6. 深色模式如何保持私密、安静、耐看？
